# StockFlow

## Project Overview

StockFlow is a Django inventory and order-reservation system with two interfaces:

- a server-rendered internal browser interface for operational users
- an authenticated Django REST Framework API

The application stores products, warehouses, inventory balances, orders, stock
movements, audit records, and reservation idempotency records in PostgreSQL.
Redis, Celery, and Celery Beat support automatic expiration of reservations.

## Main Features

- Product and warehouse catalog management
- Inventory quantities, reserved quantities, and calculated availability per
  product and warehouse
- Atomic order reservation, confirmation, cancellation, and shipping services
- Stock-in, stock-out, and absolute manual inventory adjustments
- Search, filtering, ordering, and bounded pagination in the API
- Role-based access for managers and warehouse staff
- Browser dashboard and operational pages built with Django Templates
- Scoped HTMX enhancements with full-page fallbacks
- PostgreSQL-backed reservation idempotency
- Celery-based expiration of reservations older than 30 minutes
- Read-only stock movement and audit history
- Spreadsheet-safe, streaming low-stock CSV export
- OpenAPI schema and Swagger UI generated by drf-spectacular
- pytest and pytest-django test suite
- Development and production-like Docker Compose configurations

## Architecture

StockFlow separates responsibilities by layer:

- `models.py`: database schema, relationships, indexes, and constraints
- `serializers.py` and `forms.py`: API and browser input validation
- `services.py`: transactional state changes and business workflows
- `selectors.py`: reusable, database-side read queries and annotations
- API ViewSets and browser views: HTTP coordination and permission enforcement
- Celery tasks: bounded selection and dispatch to existing domain services
- templates: server-rendered presentation, including small HTMX partials

### Why the reservation workflow is in a service layer

Reservation spans an order, all order items, multiple inventory rows, stock
movements, price snapshots, and an audit record. The service owns one
`transaction.atomic()` boundary, locks the order and inventory with
`select_for_update()`, and acquires inventory locks in deterministic product and
warehouse order. This prevents overselling and gives rollback safety: if any item
fails validation, none of the inventory, movement, order, or audit changes are
committed.

Keeping this logic outside a ViewSet or serializer also lets the API, browser
views, Celery-backed workflows, and tests reuse the same rules. It avoids
duplicating business decisions in serializers and views and keeps HTTP concerns
separate from concurrency-sensitive inventory changes.

## Database Model Summary

- **User** extends Django's `AbstractUser` with `manager` and
  `warehouse_staff` roles.
- **Product** stores name, unique SKU, category, positive decimal unit price,
  low-stock threshold, active state, and timestamps.
- **Warehouse** stores name, unique code, address, active state, and timestamps.
- **Inventory** joins one product to one warehouse and stores physical and
  reserved quantities. `available_quantity` is calculated as
  `quantity - reserved_quantity`; it is not stored separately.
- **Order** stores a unique order number, customer fields, status, total,
  `reserved_at`, and timestamps.
- **OrderItem** joins an order to a product and warehouse and stores quantity,
  a unit-price snapshot, and subtotal. A product/warehouse pair is unique within
  an order.
- **StockMovement** records inventory reservations, releases, receipts,
  deductions, and manual adjustments with an optional actor and reference.
- **AuditLog** records selected business actions, actor, target, JSON metadata,
  correlation value, and timestamp.
- **IdempotencyRecord** stores the reservation key scope, request fingerprint,
  processing state, replayable response, and expiration time.

Database constraints protect positive prices and item quantities, nonnegative
inventory, `reserved_quantity <= quantity`, unique inventory and order-item
pairs, nonnegative totals, and valid finite values for order status, stock
movement type, audit action, and idempotency status.

## Inventory Reservation Workflow

`reserve_order(order_id, performed_by, correlation_id="")` performs the
following inside one database transaction:

1. Lock the order and require `draft` status.
2. Require at least one item and positive item quantities.
3. Require active products and warehouses.
4. Sort product/warehouse keys and lock all existing inventory rows with
   `select_for_update()` in deterministic order.
5. Validate every item, including missing inventory and all insufficient-stock
   conflicts, before changing any row.
6. Increase `reserved_quantity` only; physical `quantity` does not change.
7. Refresh each item price from the current product price and recalculate
   subtotals and the order total with `Decimal`.
8. Create reservation stock movements, set `reserved_at`, transition the order
   to `reserved`, and create the audit record.

Any error rolls back the complete operation.

## Order Status Transitions

| Transition | Inventory effect | Movement |
|---|---|---|
| `draft -> reserved` | Increase reserved quantity | `reservation` |
| `reserved -> confirmed` | Decrease physical and reserved quantities | `stock_out` |
| `reserved -> cancelled` | Decrease reserved quantity only | `reservation_release` |
| `confirmed -> shipped` | No further inventory change | None |

All other transitions are rejected. Empty draft orders cannot be reserved.
Order items can be edited only while an order is in `draft` status.

## Reservation Expiration

`reserved_at` starts the 30-minute expiration window. Celery Beat runs
`apps.orders.tasks.expire_reserved_orders` every 60 seconds. The dispatcher
selects at most 100 eligible order IDs by default (the hard maximum is 500) and
queues one `expire_reserved_order` task per ID, so a failure for one order does
not prevent other queued orders from being processed.

The per-order task rechecks eligibility and calls the existing cancellation
service with `source="expiration"` and the reason
`Reservation expired after 30 minutes.` The cancellation transaction locks the
order and inventory, releases reserved stock, leaves physical stock unchanged,
and writes movement and audit records once. Duplicate execution is safe because
only a still-reserved order can transition.

Transient Django database `OperationalError` and `InterfaceError` failures use
Celery automatic retry with backoff, jitter, and at most three retries.
Unexpected exceptions fail that individual task and remain visible in worker
logs.

## Idempotency-Key Behavior

`POST /api/orders/{id}/reserve/` requires an `Idempotency-Key` header.

- Scope: authenticated user + `order_reserve` operation + key
- Fingerprint: SHA-256 of normalized method, operation, order ID, and request body
- New key: creates an `in_progress` record before reservation executes
- Same key and fingerprint after completion: replays the stored status and body
- Same unexpired key with another order or payload: returns HTTP 409
- Unexpired `in_progress`: returns HTTP 409 `IDEMPOTENCY_IN_PROGRESS`
- Unexpired `failed`: returns HTTP 409 `IDEMPOTENCY_FAILED`
- Expired record: reclaimed and treated as a new ownership attempt

Idempotency acquisition, reservation side effects, response storage, and audit
correlation run in one outer `transaction.atomic()` transaction. The unique
database constraint and row lock prevent concurrent owners from reserving stock
twice. Returned domain responses, including replayable 4xx responses, are stored
as completed responses. A processing lease lasts five minutes; a completed
response is retained for 30 days.

`delete_expired_idempotency_records()` provides bounded cleanup support, but no
periodic cleanup schedule is configured.

## Audit Logging

Audit logs and stock movements have different purposes: movements describe an
inventory change, while audit logs describe who performed a selected business
action and the affected object.

The application explicitly audits:

- product updates
- inventory adjustments
- order reservation
- order cancellation, including manual or expiration source
- order confirmation

Audit creation occurs inside the same transaction as the business change, so a
rollback leaves no success audit. Reservation API idempotency keys are stored as
the reservation audit correlation value. Audit logs are read-only in Django
Admin and in the manager browser page at `/audit-logs/`; warehouse staff cannot
access that page. The browser shows business-friendly summaries and keeps safely
escaped raw JSON in a collapsed technical-details section.

Product updates are audited when they use the browser or API update paths, which
call the product service. Product and warehouse records remain editable in
Django Admin and those direct admin edits do not create application audit logs.

## Low-Stock CSV Export

Managers can download `GET /api/reports/low-stock.csv`. This is also the Reports
link in the browser navigation.

The export is product/warehouse-level. A row is included when:

```text
quantity - reserved_quantity <= product.low_stock_threshold
```

The query and application pages reuse the same low-stock selector. The response
is a `StreamingHttpResponse`, iterates the PostgreSQL queryset in chunks, uses
`select_related()` to avoid N+1 queries, and contains product, warehouse,
quantity, availability, threshold, and generation timestamp columns. Text
starting with `=`, `+`, `-`, or `@` is prefixed with an apostrophe to reduce
spreadsheet formula interpretation, in addition to normal `csv.writer` escaping.

## HTMX Enhancement Scope

HTMX is limited to:

- inventory adjustment submission and refreshing the inventory summary and
  recent movements
- dashboard summary refresh every 60 seconds and recent movements every 120
  seconds
- reserve, confirm, cancel, and ship actions on the order detail page

The same views detect `HX-Request` and return partial templates. Standard forms,
CSRF validation, service calls, full-page POSTs, redirects, and non-HTMX fallback
remain in place. HTMX does not replace the REST API or contain business rules.

## Installation

Prerequisites for host-based development are Python, PostgreSQL, and Redis.
Docker Desktop with Docker Compose can provide all three infrastructure
services instead.

On Windows PowerShell:

```powershell
py -3 -m venv .venv
.\.venv\Scripts\python.exe -m pip install --upgrade pip
.\.venv\Scripts\python.exe -m pip install -r requirements.txt
Copy-Item .env.example .env
```

Edit the untracked `.env` file before running Django. Do not use the example
credentials or development secret in a real deployment.

## PostgreSQL Setup

StockFlow does not fall back to SQLite. Create a PostgreSQL role and database,
for example while connected as a PostgreSQL administrator:

```sql
CREATE USER stockflow WITH PASSWORD 'choose-a-local-password';
CREATE DATABASE stockflow OWNER stockflow;
```

Set `DB_NAME`, `DB_USER`, `DB_PASSWORD`, `DB_HOST`, and `DB_PORT` in `.env`, then
apply migrations:

```powershell
.\.venv\Scripts\python.exe manage.py migrate
```

## Redis and Celery Setup

For host-based operation, start Redis on the host and keep the example Redis
URLs pointed at `localhost`. In separate PowerShell terminals, run:

```powershell
.\.venv\Scripts\celery.exe -A config worker --loglevel=INFO --pool=solo
```

```powershell
.\.venv\Scripts\celery.exe -A config beat --loglevel=INFO
```

Run exactly one Beat process. The `--pool=solo` worker command is the practical
local Windows choice; Docker uses Celery's normal worker command with concurrency
2. Without both a worker and Beat, automatic reservation expiration does not run.

## Running With Docker

The development stack contains `web`, `db`, `redis`, `celery_worker`, and one
`celery_beat` service. Application services share the same image. PostgreSQL
uses the named `postgres_data` volume. Migrations and seed data are intentionally
not run automatically.

```powershell
docker compose config
docker compose up --build -d
docker compose exec web python manage.py migrate
docker compose exec web python manage.py seed_data
docker compose ps
docker compose logs celery_worker
docker compose logs celery_beat
docker compose exec web pytest
```

For a completely empty database, this ordering prevents Beat from querying
tables before migrations exist:

```powershell
docker compose build
docker compose up -d db redis
docker compose run --rm web python manage.py migrate
docker compose up -d
```

The base Compose web service uses Django `runserver` and `config.settings`; it is
for development.

## Production-like Docker Usage

`docker-compose.prod.yml` switches application services to
`config.settings_production`, runs the web process with Gunicorn, and collects
static files before startup. Supply at least a strong 50-character secret,
allowed hosts, and database credentials before resolving the configuration:

```powershell
$env:DJANGO_SECRET_KEY = "replace-with-a-random-secret-of-at-least-50-characters"
$env:DJANGO_ALLOWED_HOSTS = "stockflow.example.com"
$env:DJANGO_CSRF_TRUSTED_ORIGINS = "https://stockflow.example.com"
$env:DB_NAME = "stockflow"
$env:DB_USER = "stockflow"
$env:DB_PASSWORD = "replace-with-a-deployment-password"

docker compose -f docker-compose.yml -f docker-compose.prod.yml config
docker compose -f docker-compose.yml -f docker-compose.prod.yml up --build -d
docker compose -f docker-compose.yml -f docker-compose.prod.yml exec web python manage.py migrate
docker compose -f docker-compose.yml -f docker-compose.prod.yml exec web python manage.py check --deploy
```

Production defaults enable HTTPS redirect, secure cookies, one year of HSTS,
WhiteNoise static-file storage, persistent database connections, and optional
trusted-proxy handling. The Compose file does not provide TLS termination; put a
trusted HTTPS reverse proxy in front of it. For an isolated local diagnostic
only, HTTPS redirect and secure-cookie variables may be disabled explicitly.

The production override clears demo password variables, so running `seed_data`
there creates demo users with unusable passwords unless the deployment approach
is intentionally changed.

## Main API Endpoints

All domain API endpoints require HTTP Basic authentication or an authenticated
Django session. Warehouse staff have read access to products, warehouses,
inventory, and orders; they may adjust inventory. Manager or superuser access is
required for catalog writes, order writes/transitions, reports, and audit
visibility.

| Method | Endpoint | Behavior |
|---|---|---|
| `GET`, `POST` | `/api/products/` | List or manager create |
| `GET`, `PATCH`, `DELETE` | `/api/products/{id}/` | Read or manager mutation; `PUT` is disabled |
| `GET`, `POST` | `/api/warehouses/` | List or manager create |
| `GET`, `PATCH` | `/api/warehouses/{id}/` | Read or manager update; `PUT` and delete are disabled |
| `GET` | `/api/inventory/` | Paginated inventory list |
| `GET` | `/api/inventory/{id}/` | Inventory detail |
| `POST` | `/api/inventory/adjustments/` | Atomic stock adjustment |
| `GET` | `/api/inventory/{id}/movements/` | Paginated movements for one inventory row |
| `GET`, `POST` | `/api/orders/` | List or manager create draft |
| `GET`, `PATCH` | `/api/orders/{id}/` | Read or manager update draft; `PUT` is disabled and delete returns 405 |

## Business Rules

- Product SKU and warehouse code are unique.
- A product has at most one inventory row per warehouse.
- Physical and reserved quantities cannot be negative; reserved cannot exceed
  physical quantity.
- Available stock is always calculated as physical minus reserved stock.
- Product prices are positive decimals. Clients cannot set order status, totals,
  item price snapshots, or subtotals.
- One product/warehouse pair may appear only once in an order.
- Reservation validates every item before changing inventory and never creates
  a missing inventory row implicitly.
- Stock-out adjustments cannot reduce physical stock below reserved stock.
- `manual_adjustment` treats its quantity as the new absolute physical quantity,
  not a delta, and it must actually change the value.
- Managers manage catalog and orders. Warehouse staff can read operational data
  and adjust inventory but cannot mutate catalog records or transition orders.
- Low stock is evaluated separately for every product/warehouse inventory row.
- All application timestamps and Celery scheduling use UTC-aware datetimes.

## Error Response Format

Domain and custom API errors use this shape where the endpoint maps them:

```json
{
  "error": {
    "code": "INSUFFICIENT_STOCK",
    "message": "One or more order items do not have enough available stock.",
    "details": []
  }
}
```

Typical status codes are 400 for invalid input, 401 for missing API
authentication, 403 for role denial, 404 for a missing object, and 409 for
domain or idempotency conflicts. Some errors generated directly by Django REST
Framework, such as field validation, authentication, or default 404 responses,
retain DRF's standard `detail` or field-error shape.

## Assumptions

- PostgreSQL and Redis are external prerequisites when Docker is not used.
- Each order item reserves one product from one explicitly selected warehouse.
- Confirmation is the point at which physical stock leaves inventory; shipping
  changes only order status.
- Expiration means cancellation with source `expiration`; there is no `expired`
  order status.
- One shared inventory `reserved_quantity` is maintained per product/warehouse;
  order references in movements and order items provide the operational history.
- Superusers are treated as managers by StockFlow permission helpers.
- Production HTTPS is terminated by a trusted reverse proxy outside this
  repository.

## Known Limitations

- Reservation expiration requires Redis, a running worker, and exactly one Beat
  scheduler; it is not active when only Django is running.
- Idempotency applies only to the reserve API action. Expired idempotency records
  have a bounded cleanup helper but no scheduled cleanup task.
- The normal reservation path stores returned responses as `completed`; the
  `failed` state is handled if present but is not assigned by a dedicated
  application failure workflow.
- Audit coverage is intentionally limited to product updates, inventory
  adjustments, reservation, cancellation, and confirmation. Product creation,
  warehouse changes, draft edits, and shipping are not audited.
- Product and warehouse edits made directly in Django Admin do not call the
  application services and therefore do not create application audit records.
- There is no audit-log REST endpoint. Audit inspection is through the manager
  browser page and Django Admin.
- The Reports navigation item downloads CSV directly; there is no separate
  browser reports index.
- API authentication is Basic or session-based; token, OAuth, and SSO
  authentication are not implemented.
- Bootstrap and HTMX require access to jsDelivr. Full-page fallbacks remain
  usable without HTMX, but styling and Bootstrap behavior also depend on the CDN.
- `docker-compose.prod.yml` is production-like, not a complete deployment. It
  does not provide TLS termination, backups, centralized logs, monitoring,
  secret management, or a multi-host orchestration strategy.
- HSTS preload is disabled by default. Production `check --deploy` reports
  `security.W021` until preload is deliberately enabled for a domain whose HTTPS
  and subdomain policy is ready for browser preload lists.
- The generated OpenAPI schema currently describes Order DELETE through the
  inherited ViewSet operation even though the runtime endpoint intentionally
  returns HTTP 405.
- The unauthenticated `/health/` endpoint combines HTTP and database readiness;
  it is not a separate process-only liveness probe.

## Future Improvements

- Schedule bounded cleanup of expired idempotency records.
- Add deployment-specific TLS proxy, secret management, backups, metrics, and
  alerting.
- Add token or organization SSO authentication if external clients are needed.
- Expand audit coverage only after defining the required events and retention
  policy.
- Vendor frontend dependencies or add an approved asset build process for
  environments without CDN access.
- Add a dedicated reports page and asynchronous export only if report volume
  outgrows the current streamed CSV approach.
